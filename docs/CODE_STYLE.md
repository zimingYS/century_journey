# Century Journey 代码编写规范

## 1. 目的与适用范围

本规范约束 `src/`、`tests/`、`assets/definitions/`、构建脚本和项目文档。目标是让代码在持续扩展玩法、渲染和内容时仍然具备明确边界、稳定测试与可读的中文说明。

根目录 `AGENTS.md` 是自动化编程工具的入口；开发者和编程代理在修改代码前都必须阅读本规范。

## 2. 文件与文本

- 文件统一使用 UTF-8，换行由 Git 和编辑器按仓库设置处理。
- Rust 标识符、文件名、模块名使用英文；面向玩家的名称和提示使用中文。
- 禁止提交乱码、临时日志、编辑器元数据、存档、截图和 `target/` 产物。
- 修改既有中文文本时先确认文件编码，不使用系统默认 ANSI 编码覆盖文件。

## 3. Rust 命名与格式

- 类型、资源、组件和事件使用 `UpperCamelCase`。
- 函数、模块、变量和字段使用 `snake_case`。
- 常量使用 `SCREAMING_SNAKE_CASE`，并在名称中体现单位，例如 `_SECONDS`、`_RADIUS`。
- 事件使用完成或请求语义，例如 `BlockBrokenEvent`、`SaveRequested`；避免含糊的 `Data`、`Info`。
- 使用 `cargo fmt` 作为唯一格式标准，不手工制造与 rustfmt 冲突的排版。
- 优先消除警告，不通过扩大 `#![allow(...)]` 规避设计问题。

## 4. 中文注释规范

以下内容必须有中文文档注释：

- 模块的职责和所属层级。
- 公共结构体、枚举、资源、组件、事件和公共函数。
- 会影响存档、确定性、调度顺序、资源所有权或跨层契约的字段。
- 算法中不直观的边界条件、性能权衡与失败策略。

注释应说明“为什么”和“有什么约束”，例如：

```rust
/// 保存最近两个固定步变换，供客户端表现层进行渲染插值。
///
/// 权威 `Transform` 始终由模拟层持有，本组件不得被渲染系统反向写入。
pub struct SimulationTransformHistory {
    // ...
}
```

不要写逐行翻译式注释，例如“给变量加一”“遍历列表”。代码已经能清楚表达的内容不重复说明。

## 5. 分层与依赖方向

依赖方向遵循：

```text
Shared <- Engine/Content <- Game <- Client/App
```

- `shared`：稳定标识、跨层值类型和无运行时所有权的数据。
- `engine`：资产、任务、持久化等通用基础设施，不依赖具体玩法。
- `content`：方块、物品、配方、标签和生物群系定义，不拥有玩家状态。
- `game`：权威玩法规则、固定步模拟、存档语义和世界状态。
- `client`：输入采集、相机、渲染、UI、音效、粒子和动画表现。
- `app`：模式切换、设置、启动和各层装配。

禁止为了调用方便形成反向依赖。跨层传递优先使用事件、资源或稳定数据类型。

## 6. 模块拆分

- 一个模块只承担一个可描述的职责。
- 文件达到 500 行时检查是否混合了类型、状态机、系统注册、算法和测试等职责。
- 文件超过 800 行时，新增功能前必须先拆分；生成代码和声明式数据表除外。
- 推荐拆分为 `mod.rs` 加职责文件，例如 `animation/{types,playback,controller}.rs`。
- 拆分后通过窄范围 `pub(super)`、`pub(crate)` 或重新导出维持封装，不能把所有内部类型改成 `pub`。
- 不建立只有一层转发、没有边界价值的空模块。

## 7. Bevy ECS 与时间

- 权威移动、伤害、饥饿、掉落物物理和世界时间运行在 `FixedUpdate`。
- 相机、模型姿态、UI、音效和粒子等纯表现运行在渲染帧调度。
- 固定步与渲染帧之间通过历史快照、插值组件或稳定速度参数连接，禁止用渲染帧位置差反推模拟速度。
- 对系统顺序有要求时使用 `SystemSet`、`.before()`、`.after()` 或 `.chain()` 明确表达。
- 时间参数必须体现语义；需要真实时间时使用 `Time<Real>`，可暂停表现使用虚拟时间，固定模拟使用固定步时间。
- 游戏规则不得依赖动画关键帧、粒子或音效事件。

## 8. 错误、日志与持久化

- 可恢复错误返回 `Result`，只在不可恢复的不变量被破坏时使用 `panic!` 或 `expect`。
- 面向玩家的失败通过正式 UI 状态表达；日志用于开发诊断，不替代交互反馈。
- 日志包含领域和上下文，但不得输出令牌、个人路径或完整敏感数据。
- 存档格式变更必须增加版本和迁移测试，不直接覆盖旧字段语义。
- 随机玩法必须从确定性随机源取得种子，不能在权威模拟中直接使用非确定性线程随机数。

## 9. 测试目录规范

所有测试实现统一位于根目录 `tests/`：

```text
tests/
  unit/          # 与 src 目录结构镜像的白盒单元测试
  content/       # 内容加载与格式集成测试
  *.rs           # Cargo 自动发现的集成测试入口
```

白盒测试文件与源码镜像，例如：

```text
src/game/player/model/animation.rs
tests/unit/game/player/model/animation.rs
```

源码仅保留：

```rust
#[cfg(test)]
#[path = "../../../../tests/unit/game/player/model/animation.rs"]
mod tests;
```

- 测试名称描述行为与条件，不使用阶段编号或含糊名称。
- 修复缺陷时先添加能够复现问题的回归测试。
- 公共工作流使用集成测试；私有纯函数和边界算法使用白盒单元测试。
- 测试必须可重复，不依赖执行顺序、本机时间、随机网络和用户目录。

## 10. 数据驱动内容

- 方块、物品、配方、掉落和标签优先通过 `assets/definitions/` 扩展。
- 不在 Rust 中复制可由定义文件表达的内容属性。
- 新增标识符后同步检查纹理、模型、配方、掉落和标签引用。
- 内容格式和覆盖规则以 `docs/content-format.md` 为准。

## 11. 提交前检查清单

1. 代码职责是否放在正确层级。
2. 大文件是否继续增长，是否存在可独立命名的职责。
3. 公共契约和复杂逻辑是否有准确中文注释。
4. 是否添加了位于 `tests/` 的回归测试。
5. 是否保留确定性、存档兼容和调度顺序。
6. `cargo fmt`、`cargo check`、`cargo test`、严格 Clippy 是否全部通过。
7. 修改内容资产时，内容校验是否通过。
